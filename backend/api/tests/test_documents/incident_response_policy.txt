1. PURPOSE

This document establishes the incident response procedures and business continuity
framework for Dhiware Technologies. It ensures rapid, effective response to security
incidents and maintains critical business operations during adverse events.

2. INCIDENT RESPONSE FRAMEWORK

2.1 Incident Classification
- P1 (Critical): Data breach, ransomware, complete service outage. Response: 15 minutes.
- P2 (High): Partial service degradation, unauthorized access attempt. Response: 1 hour.
- P3 (Medium): Malware detection, policy violation. Response: 4 hours.
- P4 (Low): Phishing attempt, suspicious activity. Response: 24 hours.

2.2 Incident Response Team
- Incident Commander: CISO or designated delegate
- Technical Lead: Senior Security Engineer on-call
- Communications Lead: VP of Communications
- Legal Counsel: General Counsel (engaged for P1/P2 incidents)
- External: Mandiant retained for forensic investigation (P1 only)

2.3 Response Procedures
Phase 1 — Detection and Triage:
  - Automated detection via Splunk SIEM correlation rules
  - SOC analysts perform initial triage within 15 minutes
  - Incident ticket created in ServiceNow ITSM
  
Phase 2 — Containment:
  - Network isolation of affected systems via automated playbooks
  - Credential reset for compromised accounts
  - Preservation of forensic evidence (disk images, memory dumps)

Phase 3 — Eradication:
  - Root cause analysis performed by Security Engineering
  - Malware removal and system reimaging
  - Vulnerability patching that enabled the incident

Phase 4 — Recovery:
  - Systems restored from verified clean backups
  - Gradual service restoration with monitoring
  - User access re-provisioned after verification

Phase 5 — Post-Incident:
  - Post-mortem report within 5 business days
  - Lessons learned documented and shared
  - Control improvements implemented within 30 days

3. BUSINESS CONTINUITY

3.1 Recovery Objectives
- Recovery Time Objective (RTO): 4 hours for critical systems
- Recovery Point Objective (RPO): 1 hour (maximum data loss tolerance)
- Maximum Tolerable Downtime (MTD): 24 hours

3.2 Backup Strategy
- Database: Continuous replication to DR site (AWS us-west-2)
- Application: Container images stored in ECR with multi-region replication
- Configuration: Infrastructure-as-Code in Git (immutable deployments)
- Backup verification: Automated restore tests run weekly

3.3 Disaster Recovery
- Primary: AWS ap-south-1 (Mumbai)
- DR Site: AWS us-west-2 (Oregon)
- Failover: Automated via Route 53 health checks
- DR drills: Conducted quarterly with full failover simulation
- Last successful DR test: December 12, 2024

4. COMMUNICATION PLAN

4.1 Internal Notification
- P1: All-hands alert via PagerDuty + Slack #incident-war-room
- P2: Security team + affected service owners
- P3/P4: Security team notification only

4.2 External Notification
- Regulatory: Within 72 hours per GDPR/DPDPA requirements
- Customers: Within 48 hours for data breaches affecting customer data
- Law enforcement: Per legal counsel guidance for criminal activity

5. COMPLIANCE MAPPING

This policy addresses:
- ISO 27001:2022 — A.16.1.1 (Incident management responsibilities)
- ISO 27001:2022 — A.17.1.1 (Information security continuity planning)
- SOC 2 — A1.2 (Availability — environmental protections)

6. TRAINING AND AWARENESS

- All employees complete annual incident response awareness training.
- Tabletop exercises conducted quarterly with the incident response team.
- Phishing simulation campaigns run monthly (target: <5% click rate).
- Last training completion rate: 97% (Q4 2024).
